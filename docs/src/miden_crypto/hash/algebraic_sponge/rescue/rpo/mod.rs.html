14/// Implementation of the Rescue Prime Optimized hash function with 256-bit output.
15///
16/// The hash function is implemented according to the Rescue Prime Optimized
17/// [specifications](https://eprint.iacr.org/2022/1577) while the padding rule follows the one
18/// described [here](https://eprint.iacr.org/2023/1045).
19///
20/// The parameters used to instantiate the function are:
21/// * Field: 64-bit prime field with modulus p = 2^64 - 2^32 + 1.
22/// * State width: 12 field elements.
23/// * Rate size: r = 8 field elements.
24/// * Capacity size: c = 4 field elements.
25/// * Number of founds: 7.
26/// * S-Box degree: 7.
27///
28/// The above parameters target a 128-bit security level. The digest consists of four field elements
29/// and it can be serialized into 32 bytes (256 bits).
30///
31/// ## Hash output consistency
32/// Functions [hash_elements()](Rpo256::hash_elements), [merge()](Rpo256::merge), and
33/// [merge_with_int()](Rpo256::merge_with_int) are internally consistent. That is, computing
34/// a hash for the same set of elements using these functions will always produce the same
35/// result. For example, merging two digests using [merge()](Rpo256::merge) will produce the
36/// same result as hashing 8 elements which make up these digests using
37/// [hash_elements()](Rpo256::hash_elements) function.
38///
39/// However, [hash()](Rpo256::hash) function is not consistent with functions mentioned above.
40/// For example, if we take two field elements, serialize them to bytes and hash them using
41/// [hash()](Rpo256::hash), the result will differ from the result obtained by hashing these
42/// elements directly using [hash_elements()](Rpo256::hash_elements) function. The reason for
43/// this difference is that [hash()](Rpo256::hash) function needs to be able to handle
44/// arbitrary binary strings, which may or may not encode valid field elements - and thus,
45/// deserialization procedure used by this function is different from the procedure used to
46/// deserialize valid field elements.
47///
48/// Thus, if the underlying data consists of valid field elements, it might make more sense
49/// to deserialize them into field elements and then hash them using
50/// [hash_elements()](Rpo256::hash_elements) function rather than hashing the serialized bytes
51/// using [hash()](Rpo256::hash) function.
52///
53/// ## Domain separation
54/// [merge_in_domain()](Rpo256::merge_in_domain) hashes two digests into one digest with some domain
55/// identifier and the current implementation sets the second capacity element to the value of
56/// this domain identifier. Using a similar argument to the one formulated for domain separation of
57/// the RPX hash function in Appendix C of its [specification](https://eprint.iacr.org/2023/1045),
58/// one sees that doing so degrades only pre-image resistance, from its initial bound of c.log_2(p),
59/// by as much as the log_2 of the size of the domain identifier space. Since pre-image resistance
60/// becomes the bottleneck for the security bound of the sponge in overwrite-mode only when it is
61/// lower than 2^128, we see that the target 128-bit security level is maintained as long as
62/// the size of the domain identifier space, including for padding, is less than 2^128.
63///
64/// ## Hashing of empty input
65/// The current implementation hashes empty input to the zero digest [0, 0, 0, 0]. This has
66/// the benefit of requiring no calls to the RPO permutation when hashing empty input.
67#[allow(rustdoc::private_intra_doc_links)]
68#[derive(Debug, Copy, Clone, Eq, PartialEq)]
69pub struct Rpo256();

163    // RESCUE PERMUTATION
164    // --------------------------------------------------------------------------------------------

166    /// Applies RPO permutation to the provided state.
167    #[inline(always)]
168    pub fn apply_permutation(state: &mut [Felt; STATE_WIDTH]) {
169        for i in 0..NUM_ROUNDS {
170            Self::apply_round(state, i);
171        }
172    }
173
174    /// RPO round function.
175    #[inline(always)]
176    pub fn apply_round(state: &mut [Felt; STATE_WIDTH], round: usize) {
177        // apply first half of RPO round
178        apply_mds(state);
179        if !add_constants_and_apply_sbox(state, &ARK1[round]) {
180            add_constants(state, &ARK1[round]);
181            apply_sbox(state);
182        }
183
184        // apply second half of RPO round
185        apply_mds(state);
186        if !add_constants_and_apply_inv_sbox(state, &ARK2[round]) {
187            add_constants(state, &ARK2[round]);
188            apply_inv_sbox(state);
189        }
190    }
191}

213/// Plonky3-compatible RPO permutation.
214///
215/// This struct wraps the RPO256 permutation and implements Plonky3's `Permutation` and
216/// `CryptographicPermutation` traits, allowing RPO to be used within the Plonky3 ecosystem.
217///
218/// The permutation operates on a state of 12 field elements (STATE_WIDTH = 12), with:
219/// - Rate: 8 elements (positions 0-7)
220/// - Capacity: 4 elements (positions 8-11)
221/// - Digest output: 4 elements (positions 0-3)
222#[derive(Debug, Copy, Clone, Eq, PartialEq)]
223pub struct RpoPermutation256;

231

233    /// the remaining 4 elements are reserved for capacity.
234    pub const STATE_WIDTH: usize = STATE_WIDTH;

237    pub const RATE_RANGE: Range<usize> = Rpo256::RATE_RANGE;
238
239    /// The capacity portion of the state is located in elements 8, 9, 10, and 11.
240    pub const CAPACITY_RANGE: Range<usize> = Rpo256::CAPACITY_RANGE;
241

272/// RPO-based hasher using Plonky3's PaddingFreeSponge.
273///
274/// This provides a sponge-based hash function with:
275/// - WIDTH: 12 field elements (total state size)
276/// - RATE: 8 field elements (input/output rate)
277/// - OUT: 4 field elements (digest size)
278pub type RpoHasher = PaddingFreeSponge<RpoPermutation256, 12, 8, 4>;
279
280/// RPO-based compression function using Plonky3's TruncatedPermutation.
281///
282/// This provides a 2-to-1 compression function for Merkle tree construction with:
283/// - CHUNK: 2 (number of input chunks - i.e., 2 digests of 4 elements each = 8 elements)
284/// - N: 4 (output size in field elements)
285/// - WIDTH: 12 (total state size)
286///
287/// The compression function takes 8 field elements (2 digests) as input and produces
288/// 4 field elements (1 digest) as output.
289pub type RpoCompression = TruncatedPermutation<RpoPermutation256, 2, 4, 12>;
290
291/// RPO-based challenger using Plonky3's DuplexChallenger.
292///
293/// This provides a Fiat-Shamir transform implementation for interactive proof protocols,
294/// with:
295/// - F: Generic field type (typically the same as Felt)
296/// - WIDTH: 12 field elements (sponge state size)
297/// - RATE: 8 field elements (rate of absorption/squeezing)
298pub type RpoChallenger<F> = DuplexChallenger<F, RpoPermutation256, 12, 8>;
