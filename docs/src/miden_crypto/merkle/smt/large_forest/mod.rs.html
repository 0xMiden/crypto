1//! A high-performance sparse merkle tree forest with pluggable backends.
2//!
3//! # Semantic Layout
4//!
5//! Much like `SparseMerkleTree`, the forest stores trees of depth 64 that use the compact leaf
6//! optimization to uniquely store 256-bit elements. This reduces both the size of a merkle path,
7//! and the computational work necessary to perform queries into the trees.
8//!
9//! # Storing Trees and Versions
10//!
11//! The usage of an SMT forest is conceptually split into two parts: a collection that is able to
12//! store **multiple, unrelated trees**, and a container for **multiple versions of those trees**.
13//! Both of these use-cases are supported by the forest, but have an explicit delineation between
14//! them in both the API and the implementation. This has two impacts that a client of the forest
15//! must understand.
16//!
17//! - While, when using a [`Backend`] that can persist data, **only the current full tree state is
18//!   persisted**, while **the historical data will not be**. This is designed into the structure of
19//!   the forest, and does not depend on the choice of storage backend.
20//! - It is more expensive to query a given tree at an older point in its history than it is to
21//!   query it at a newer point, and querying at the current tree will always take the least time.
22//!
23//! # Lineages
24//!
25//! We term a set of trees where each is derived from the previous version to be a **lineage**. A
26//! single lineage semantically contains the **full information** on the current state of the tree,
27//! alongside a set of deltas which describe how to change that full tree to return to a historical
28//! state of that tree.
29//!
30//! While any given [`Backend`] may choose to share data between lineages, this behavior is not
31//! guaranteed, and must not be relied upon.
32//!
33//! # Tree Identification
34//!
35//! It is possible for multiple lineages to contain a tree with identical leaves and hence an
36//! identical root. As we store lineages separately, we need some way to specify which instance of a
37//! given root we mean.
38//!
39//! This is done by identifying trees using the [`TreeId`], which combines the tree's root value
40//! with a user-provided identifier that tags the tree with a 'domain'. This allows distinguishing
41//! between otherwise identical trees. Users must take care to ensure that each domain is unique, as
42//! reusing them will result in overwriting data in the wrong domain, and that queries may return
43//! incorrect results.
44//!
45//! # Data Storage
46//!
47//! The SMT forest is parametrized over the [`Backend`] implementation that it uses. These backends
48//! may have significantly varied performance characteristics, and hence any performance analysis of
49//! the forest should be done in conjunction with a specific backend. The forest itself takes pains
50//! to not make any assumptions about properties of the backend in use.
51//!
52//! Take care to read the documentation of the specific [`Backend`] that you are planning to use in
53//! order to understand its performance, gotchas, and other such details.

83/// A high-performance forest of sparse merkle trees with pluggable storage.
84///
85/// # Current and Frozen Trees
86///
87/// Trees in the forest fall into two categories:
88///
89/// 1. **Current:** These trees represent the latest version of their 'tree lineage' and can be
90///    modified to generate a new tree version in the forest.
91/// 2. **Frozen:** These are historical versions of trees that are no longer current, and are
92///    considered 'frozen' and hence cannot be modified to generate a new tree version in the
93///    forest. This is because being able to do so would effectively create a "fork" in the history,
94///    and hence allow the forest to yield potentially invalid responses with regard to the
95///    blockchain history.
96///
97/// The API is designed to avoid any possibility of modifying frozen trees in the forest, and hence
98/// ensure the correctness of the history stored in the forest.
99///
100/// # Performance
101///
102/// The performance characteristics of this forest depend heavily on the choice of underlying
103/// [`Backend`] implementation. Where something more specific can be said about a particular method
104/// call, the documentation for that method will state it.
105#[allow(dead_code)] // Temporarily
106#[derive(Debug)]
107pub struct LargeSmtForest<B: Backend> {
108    /// The backend for storing the full trees that exist as part of the forest. It makes no
109    /// guarantees as to where the tree data is stored, and **must not be exposed** in the API of
110    /// the forest for correctness.
111    backend: B,
112
113    /// The container for the in-memory data associated with each lineage in the forest.
114    ///
115    /// It must contain an entry for every tree lineage in the forest.
116    lineage_data: Map<LineageId, LineageData>,
117
118    /// A set tracking which lineage which lineages have histories containing actual deltas in
119    /// order to speed up querying.
120    ///
121    /// It must always be maintained as a strict subset of `lineage_data.keys()`.
122    non_empty_histories: Set<LineageId>,
123}

157/// These methods provide the ability to perform basic operations on the forest without the need to
158/// query the backend.
159///
160/// # Performance
161///
162/// All of these methods can be performed fully in-memory, and hence their performance is
163/// predictable on a given machine regardless of the choice of [`Backend`] instance being used by
164/// the forest.
165impl<B: Backend> LargeSmtForest<B> {
166    /// Returns an iterator that yields all the (uniquely identified) roots that the forest knows
167    /// about, including those from historical versions.
168    ///
169    /// The iteration order of these roots is unspecified.
170    pub fn roots(&self) -> impl Iterator<Item = UniqueRoot> {
171        // As the history container does not deal in roots with domains, we have to attach the
172        // corresponding domain to each root, and do this as lazily as possible to avoid
173        // materializing more things than we need to.
174        self.lineage_data
175            .iter()
176            .flat_map(|(l, d)| d.roots().map(|r| UniqueRoot::new(*l, r)))
177    }
178
179    /// Gets the latest version of the tree for the provided `lineage`, if that lineage is in the
180    /// forest, or returns [`None`] otherwise.
181    pub fn latest_version(&self, lineage: LineageId) -> Option<VersionId> {
182        self.lineage_data.get(&lineage).map(|d| d.latest_version)
183    }
184
185    /// Returns an iterator that yields the root values for trees within the specified `lineage`, or
186    /// [`None`] if the lineage is not known.
187    ///
188    /// The iteration order of the roots is guaranteed to move backward in time, with earlier items
189    /// being roots from versions closer to the present. The current root of the lineage will always
190    /// be the first item yielded by the iterator.
191    pub fn lineage_roots(&self, lineage: LineageId) -> Option<impl Iterator<Item = RootValue>> {
192        self.lineage_data.get(&lineage).map(|d| d.roots())
193    }
194
195    /// Gets the value root of the newest tree in the provided `lineage`, if that lineage is in the
196    /// forest, or returns [`None`] otherwise.
197    pub fn latest_root(&self, lineage: LineageId) -> Option<RootValue> {
198        self.lineage_data.get(&lineage).map(|d| d.latest_root)
199    }
200
201    /// Returns an iterator that yields the historical root values for trees within the specified
202    /// `lineage`, or [`None`] if the lineage is not known.
203    ///
204    /// The iteration order of the roots is guaranteed to move backward in time, with earlier items
205    /// being roots from versions closer to the present. It does _not_ include the latest root in
206    /// the specified `lineage`.
207    pub fn historical_roots(&self, lineage: LineageId) -> Option<impl Iterator<Item = RootValue>> {
208        // We skip the first element as this is always guaranteed to be the current root for the
209        // lineage.
210        self.lineage_roots(lineage).map(|i| i.skip(1))
211    }
212
213    /// Returns the number of trees in the forest that have unique identity.
214    ///
215    /// This is **not** the number of unique tree lineages in the forest, as it includes all
216    /// historical trees as well. For that, see [`Self::lineage_count`].
217    pub fn tree_count(&self) -> usize {
218        self.roots().count()
219    }
220
221    /// Returns the number of unique tree lineages in the forest.
222    ///
223    /// This is **not** the number of unique trees in the forest, as it does not include all
224    /// versions in each lineage. For that, see [`Self::tree_count`].
225    pub fn lineage_count(&self) -> usize {
226        self.lineage_data.len()
227    }
228
229    /// Returns data describing what information the forest knows about the provided `root`.
230    pub fn root_info(&self, root: TreeId) -> RootInfo {
231        if let Some(d) = self.lineage_data.get(&root.lineage()) {
232            if d.latest_version == root.version() {
233                RootInfo::LatestVersion(d.latest_root)
234            } else {
235                match d.history.root_for_version(root.version()) {
236                    Ok(r) => RootInfo::HistoricalVersion(r),
237                    Err(_) => RootInfo::Missing,
238                }
239            }
240        } else {
241            RootInfo::Missing
242        }
243    }
244
245    /// Removes all tree versions in the forest that are older than the provided `version`, but
246    /// always retains the latest tree in each lineage.
247    pub fn truncate(&mut self, version: VersionId) {
248        let mut newly_empty = Set::default();
249
250        self.non_empty_histories.iter().for_each(|l| {
251            if let Some(d) = self.lineage_data.get_mut(l)
252                && d.truncate(version)
253            {
254                newly_empty.insert(*l);
255            }
256        });
257
258        self.non_empty_histories.extend(newly_empty);
259    }
260}

405// LINEAGE DATA
406// ================================================================================================
407
408/// The data that the forest stores in memory for each lineage of trees.
409#[derive(Clone, Debug)]
410struct LineageData {
411    /// The historical overlays for the lineage.
412    pub history: History,
413
414    /// The version associated with the latest tree in the lineage.
415    pub latest_version: VersionId,
416
417    /// The value of the root for the latest tree in the lineage.
418    pub latest_root: RootValue,
419}
420
421impl LineageData {
422    /// Gets an iterator that yields each root in the lineage.
423    ///
424    /// The iteration order of the roots is guaranteed to move backward in time, with earlier items
425    /// being roots from versions closer to the present. The current root of the lineage will always
426    /// be the first item yielded by the iterator.
427    fn roots(&self) -> impl Iterator<Item = RootValue> {
428        once(self.latest_root).chain(self.history.roots())
429    }
430
431    /// Truncates the information on this tree to the provided `version`, returning `true` if the
432    /// history is empty after truncation, and `false` otherwise.
433    ///
434    /// In the case that the version of the latest tree in the lineage is older than `version`, this
435    /// current version is always retained.
436    pub(super) fn truncate(&mut self, version: VersionId) -> bool {
437        if version >= self.latest_version {
438            // Truncation in the history is defined such that it never removes a version that could
439            // possibly serve as the latest delta for a newer version. This is because it cannot
440            // safely know if a version `v` is between the latest delta `d` and the current version
441            // `c`, as it has no knowledge of the current version.
442            //
443            // Thus, if we have a version `v` such that `d <= v < c`, we need to retain the
444            // reversion delta `d` in the history to correctly service queries for `v`. If, however,
445            // we have `d < c <= v` we need to explicitly remove the last delta as well.
446            //
447            // To that end, we handle the latter case first, by explicitly calling
448            // `History::clear()`.
449            self.history.clear();
450            true
451        } else {
452            // The other case is `v < c`, which is handled simply by the truncation mechanism in the
453            // history as we want. In other words, it retains the necessary delta, and so we can
454            // just call it here.
455            self.history.truncate(version);
456            false
457        }
458    }
