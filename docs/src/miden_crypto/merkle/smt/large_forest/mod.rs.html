1//! A high-performance sparse merkle tree forest with pluggable backends.
2//!
3//! # Semantic Layout
4//!
5//! Much like the `SparseMerkleTree`, the forest stores its trees at depth [`SMT_DEPTH`] and then
6//! relies on the compact leaf optimization to uniquely store the 256-bit elements that it contains.
7//! This is done to both reduce the size of a merkle path, and to reduce the computational work
8//! necessary to perform queries into the trees.
9//!
10//! It also has the benefit of significantly reducing the memory usage for the forest. Even in cases
11//! where it relies on a persistent backend, the other peripheral structures are able to be smaller
12//! and thus use less memory.
13//!
14//! # Backends
15//!
16//! The forest is implemented to rely on the API and contract conformance of an arbitrary
17//! [`Backend`] implementation. These backends provide the storage for full trees in the forest, and
18//! are the main extension point for the way the forest functions.
19//!
20//! The [`InMemoryBackend`] provides simple, in-memory storage for the full trees in the forest. It
21//! is _primarily_ intended to be used for testing purposes, but should nevertheless be correct and
22//! functional for production use-cases if no persistence is required.
23//!
24//! While any given [`Backend`] may choose to share data between lineages, this behavior is not
25//! guaranteed, and must not be relied upon.
26//!
27//! ## Performance
28//!
29//! Each [`Backend`] provides the same set of functionality to the forest, but may exhibit
30//! significant variance in their performance characteristics. As a result, **any performance
31//! analysis of the forest should be done in conjunction with a specific backend**.
32//!
33//! Take care to read the documentation of the specific [`Backend`] that you are planning to use in
34//! order to understand its performance, potential gotchas, and other such details.
35//!
36//! # Storing Trees and Versions
37//!
38//! An SMT forest conceptually performs two roles. Firstly, it acts as a collection that is able to
39//! store **multiple, unrelated trees**. Secondly, it is a container for **multiple versions of a
40//! given tree**. In order to make it tractable to implement a performant forest with pluggable
41//! backends, this type makes an explicit delineation between these use-cases in both the API and
42//! the implementation.
43//!
44//! ## Lineages
45//!
46//! We term a set of trees, where each tree is derived from changing the previous version, to be a
47//! **lineage** of trees. A single lineage contains the information necessary to reconstruct any
48//! previous version of the tree, within the bounds of the history that the forest stores.
49//!
50//! Users must take care to ensure that each lineage identifier is unique, as reuse of these
51//! identifiers can result in data corruption and hence queries that return incorrect results.
52//!
53//! # Tree Identification
54//!
55//! It is possible for a tree with identical leaves (and hence an identical root) to exist in
56//! multiple lineages in the forest. As lineages are stored separately, there needs to be a way to
57//! specify the precise instance of a given tree.
58//!
59//! Trees are thus identified using the [`TreeId`], which combines the **lineage** in which the tree
60//! exists with the **version** in that lineage.
61//!
62//! ## Potential Gotchas
63//!
64//! The separation of the forest into lineages of trees has a few impacts that a client of the
65//! forest must understand:
66//!
67//! - When using a [`Backend`] that offers data persistence, **only the state of the current version
68//!   of each lineage is persisted**, while **the historical data is not persisted**. This is part
69//!   of the way the forest is structured, and does not depend on the choice of backend.
70//! - It is always going to be more expensive to query a given lineage at **an older point** in its
71//!   history than it is to query at a newer point.
72//! - Querying **the latest tree in a lineage will take the least time**.
73//!
74//! # Batch Operations
75//!
76//! The [`LargeSmtForest::update_tree`] and [`LargeSmtForest::update_forest`] methods are what is
77//! known as **batch operations**. In other words, they are performed in one go and only produce a
78//! one-stage update to the forest, rather than a sequence of updates.
79//!
80//! These methods should be used wherever possible (especially preferring `update_forest` over a
81//! sequence of `update_tree` calls) as this will allow the forest and its backend to exploit as
82//! much parallelism as possible in the updates.
83//!
84//! # Examples
85//!
86//! The following section contains usage examples for the forest. They rely on the included
87//! [`InMemoryBackend`] for simplicity, but will work with any conformant [`Backend`]
88//! implementation. Each example is designed to build upon the last.
89//!
90//! ## Constructing a Forest
91//!
92//! A new forest can be constructed by calling either [`LargeSmtForest::new`], which will use a
93//! default [`Config`], or by explicitly providing the config in [`LargeSmtForest::with_config`].
94//!
95//! ```
96//! use miden_crypto::merkle::smt::{ForestInMemoryBackend, LargeSmtForest};
97//! # use miden_crypto::merkle::smt::LargeSmtForestError;
98//! #
99//! # fn main() -> Result<(), LargeSmtForestError> {
100//!
101//! let backend = ForestInMemoryBackend::new();
102//! let forest = LargeSmtForest::new(backend)?;
103//! #
104//! # Ok(())
105//! # }
106//! ```
107//!
108//! Upon startup, the forest has to read the lineages it knows from the provided storage. If it
109//! cannot get this information, it cannot start up properly and the constructor may return an
110//! error.
111//!
112//! ## Adding a Lineage
113//!
114//! Each tree in the forest belongs to a _lineage_, identified by a [`LineageId`]. In order to work
115//! with a lineage in the forest, that lineage first has to be added to it! Adding a lineage can
116//! either add the empty tree, or specify a set of modifications on the empty tree to create a
117//! starting state.
118//!
119//! ```
120//! # use miden_crypto::merkle::smt::LargeSmtForestError;
121//! # use miden_crypto::merkle::smt::{ForestInMemoryBackend, LargeSmtForest};
122//! use miden_crypto::{
123//!     Word,
124//!     merkle::smt::{LineageId, SmtUpdateBatch},
125//! };
126//!
127//! # fn main() -> Result<(), LargeSmtForestError> {
128//! # let backend = ForestInMemoryBackend::new();
129//! # let mut forest = LargeSmtForest::new(backend)?;
130//! #
131//! // We can just make some arbitrary values here for demonstration.
132//! let key_1 = Word::parse("0x42").unwrap();
133//! let value_1 = Word::parse("0x80").unwrap();
134//! let key_2 = Word::parse("0xAB").unwrap();
135//! let value_2 = Word::parse("0xCD").unwrap();
136//!
137//! // Operations are most cleanly specified using a builder.
138//! let mut operations = SmtUpdateBatch::empty();
139//! operations.add_insert(key_1, value_1);
140//! operations.add_insert(key_2, value_2);
141//!
142//! // To add a new lineage we also need to give it a lineage ID, and a version.
143//! let lineage = LineageId::new([0x42; 32]);
144//! let version_1 = 1;
145//!
146//! // Now we can add the lineage to the forest!
147//! assert!(forest.add_lineage(lineage, version_1, operations).is_ok());
148//! #
149//! # Ok(())
150//! # }
151//! ```
152//!
153//! ## Modifying a Lineage
154//!
155//! A forest is not all that useful if we cannot update it! Modifying a lineage is much like adding
156//! a new one, in that we specify operations to be performed on the latest tree in that lineage.
157//!
158//! ```
159//! # use miden_crypto::merkle::smt::LargeSmtForestError;
160//! # use miden_crypto::{
161//! #     Word,
162//! #     merkle::smt::{ForestInMemoryBackend, LargeSmtForest, LineageId, SmtUpdateBatch},
163//! # };
164//! #
165//! # fn main() -> Result<(), LargeSmtForestError> {
166//! # let backend = ForestInMemoryBackend::new();
167//! # let mut forest = LargeSmtForest::new(backend)?;
168//! #
169//! # // We can just make some arbitrary values here for demonstration.
170//! # let key_1 = Word::parse("0x42").unwrap();
171//! # let value_1 = Word::parse("0x80").unwrap();
172//! # let key_2 = Word::parse("0xAB").unwrap();
173//! # let value_2 = Word::parse("0xCD").unwrap();
174//! #
175//! # // Operations are most cleanly specified using a builder.
176//! # let mut operations = SmtUpdateBatch::empty();
177//! # operations.add_insert(key_1, value_1);
178//! # operations.add_insert(key_2, value_2);
179//! #
180//! # // To add a new lineage we also need to give it a lineage ID, and a version.
181//! # let lineage = LineageId::new([0x42; 32]);
182//! # let version_1 = 1;
183//! #
184//! # // Now we can add the lineage to the forest!
185//! # forest.add_lineage(lineage, version_1, operations)?;
186//! #
187//! // Let's make another arbitrary value.
188//! let key_3 = Word::parse("0x67").unwrap();
189//! let value_3 = Word::parse("0x96").unwrap();
190//!
191//! // And build a batch of operations again.
192//! let mut operations = SmtUpdateBatch::empty();
193//! operations.add_insert(key_3, value_3);
194//! operations.add_remove(key_1);
195//!
196//! // Now we can simply update the tree all in one go with our changes.
197//! let version_2 = version_1 + 1;
198//! assert!(forest.update_tree(lineage, version_2, operations).is_ok());
199//! #
200//! # Ok(())
201//! # }
202//! ```
203//!
204//! Multiple lineages can be modified at once using the [`LargeSmtForest::update_forest`] method,
205//! which works very similarly to the [`LargeSmtForest::update_tree`] method shown above.
206//!
207//! ## Querying a Lineage
208//!
209//! Modification is just one part of the puzzle, however. It is just as important to be able to get
210//! data _out_ of the forest too!
211//!
212//! ```
213//! # use miden_crypto::merkle::smt::LargeSmtForestError;
214//! # use miden_crypto::{
215//! #     Word,
216//! #     merkle::smt::{ForestInMemoryBackend, LargeSmtForest, LineageId, SmtUpdateBatch},
217//! # };
218//! use miden_crypto::merkle::smt::{TreeEntry, TreeId};
219//!
220//! # fn main() -> Result<(), LargeSmtForestError> {
221//! # let backend = ForestInMemoryBackend::new();
222//! # let mut forest = LargeSmtForest::new(backend)?;
223//! #
224//! # // We can just make some arbitrary values here for demonstration.
225//! # let key_1 = Word::parse("0x42").unwrap();
226//! # let value_1 = Word::parse("0x80").unwrap();
227//! # let key_2 = Word::parse("0xAB").unwrap();
228//! # let value_2 = Word::parse("0xCD").unwrap();
229//! #
230//! # // Operations are most cleanly specified using a builder.
231//! # let mut operations = SmtUpdateBatch::empty();
232//! # operations.add_insert(key_1, value_1);
233//! # operations.add_insert(key_2, value_2);
234//! #
235//! # // To add a new lineage we also need to give it a lineage ID, and a version.
236//! # let lineage = LineageId::new([0x42; 32]);
237//! # let version_1 = 1;
238//! #
239//! # // Now we can add the lineage to the forest!
240//! # forest.add_lineage(lineage, version_1, operations)?;
241//! #
242//! # // Let's make another arbitrary value.
243//! # let key_3 = Word::parse("0x67").unwrap();
244//! # let value_3 = Word::parse("0x96").unwrap();
245//! #
246//! # // And build a batch of operations again.
247//! # let mut operations = SmtUpdateBatch::empty();
248//! # operations.add_insert(key_3, value_3);
249//! # operations.add_remove(key_1);
250//! #
251//! # // Now we can simply update the tree all in one go with our changes.
252//! # let version_2 = version_1 + 1;
253//! # forest.update_tree(lineage, version_2, operations)?;
254//! #
255//! // As discussed above, trees are identified by a combination of their lineage and version.
256//! let old_tree = TreeId::new(lineage, version_1);
257//! let current_tree = TreeId::new(lineage, version_2);
258//!
259//! // The first really useful query is `open`, which gets the opening for the specified key. We can
260//! // get openings for the current tree AND the historical trees.
261//! assert!(forest.open(old_tree, key_1).is_ok());
262//! assert!(forest.open(current_tree, key_3).is_ok());
263//!
264//! // We can also just `get` the value associated with a key, which returns `None` if the key is
265//! // not populated.
266//! assert_eq!(forest.get(old_tree, key_1)?, Some(value_1));
267//! assert_eq!(forest.get(current_tree, key_3)?, Some(value_3));
268//! assert!(forest.get(current_tree, key_1)?.is_none());
269//!
270//! // We can also get an iterator over all the entries in the tree.
271//! let entries_old: Vec<_> = forest.entries(old_tree)?.collect();
272//! let entries_current: Vec<_> = forest.entries(current_tree)?.collect();
273//! assert!(entries_old.contains(&TreeEntry { key: key_1, value: value_1 }));
274//! assert!(entries_old.contains(&TreeEntry { key: key_2, value: value_2 }));
275//! assert!(!entries_old.contains(&TreeEntry { key: key_3, value: value_3 }));
276//! assert!(!entries_current.contains(&TreeEntry { key: key_1, value: value_1 }));
277//! assert!(entries_current.contains(&TreeEntry { key: key_2, value: value_2 }));
278//! assert!(entries_current.contains(&TreeEntry { key: key_3, value: value_3 }));
279//! #
280//! # Ok(())
281//! # }
282//! ```
283//!
284//! There are many other kinds of queries of course, so taking a look at the methods available on
285//! [`LargeSmtForest`] is a good starting point.

327/// A high-performance forest of sparse merkle trees with pluggable storage backends.
328///
329/// See the module documentation for more information.
330#[derive(Debug)]
331pub struct LargeSmtForest<B: Backend> {
332    /// The configuration for how the forest functions.
333    config: Config,
334
335    /// The backend for storing the full trees that exist as part of the forest.
336    ///
337    /// It makes no guarantees as to where the tree data is stored, and **must not be exposed** in
338    /// the API of the forest to ensure that internal invariants are maintained.
339    backend: B,
340
341    /// The container for the in-memory data associated with each lineage in the forest.
342    ///
343    /// It must contain an entry for every tree lineage in the forest.
344    lineage_data: Map<LineageId, LineageData>,
345
346    /// A set tracking which lineage which lineages have histories containing actual deltas in
347    /// order to speed up querying.
348    ///
349    /// It must always be maintained as a strict subset of `lineage_data.keys()`.
350    non_empty_histories: Set<LineageId>,
351}

356/// These functions deal with the creation of new forest instances, and hence rely on the ability to
357/// query the backend to do so.
358///
359/// # Performance
360///
361/// All the methods in this impl block require access to the underlying [`Backend`] instance to
362/// return results. This means that their performance will depend heavily on the specific instance
363/// with which the forest was constructed.
364///
365/// Where anything more specific can be said about performance, the method documentation will
366/// contain more detail.
367impl<B: Backend> LargeSmtForest<B> {
368    /// Constructs a new forest backed by the provided `backend` using the default [`Config`] for
369    /// the forest's behavior.
370    ///
371    /// This constructor will treat whatever state is contained within the provided `backend` as the
372    /// starting state for the forest. This means that, if you pass a newly-initialized storage, the
373    /// forest will start in an empty state. Similarly, if you pass a `backend` that already
374    /// contains some data (loaded from disk, for example), then the forest will start in that state
375    /// instead.
376    ///
377    /// # Performance
378    ///
379    /// For performance notes on this method, see [`Self::with_config`] instead.
380    ///
381    /// # Errors
382    ///
383    /// - [`LargeSmtForestError::Other`] if the forest cannot be started up correctly using the
384    ///   provided `backend`.
385    pub fn new(backend: B) -> Result<Self> {
386        Self::with_config(backend, Config::default())
387    }
388
389    /// Constructs a new forest backed by the provided `backend` and configuring behavior using the
390    /// provided `config`.
391    ///
392    /// This constructor will treat whatever state is contained within the provided `backend` as the
393    /// starting state for the forest. This means that, if you pass a newly-initialized storage, the
394    /// forest will start in an empty state. Similarly, if you pass a `backend` that already
395    /// contains some data (loaded from disk, for example), then the forest will start in that state
396    /// instead.
397    ///
398    /// # Performance
399    ///
400    /// This method is required to load the basic tree metadata from the backend during forest
401    /// construction. This metadata should be stored separately, and hence this method should take a
402    /// relatively small amount of time.
403    ///
404    /// # Errors
405    ///
406    /// - [`LargeSmtForestError::Fatal`] if the forest cannot be started up correctly using the
407    ///   provided `backend`.
408    pub fn with_config(backend: B, config: Config) -> Result<Self> {
409        // The lineages at initialization time are whichever ones the backend knows about. To that
410        // end, we read from the backend and construct the starting state for each known lineage.
411        let lineage_data = backend
412            .trees()?
413            .map(|t| {
414                let data = LineageData {
415                    history: History::empty(config.max_history_versions()),
416                    latest_version: t.version(),
417                    latest_root: t.root(),
418                };
419                (t.lineage(), data)
420            })
421            .collect::<Map<LineageId, LineageData>>();
422
423        // As no backend is able to preserve history, we can unconditionally initialize the tracking
424        // for non-empty histories as empty.
425        let non_empty_histories = Set::default();
426
427        Ok(Self {
428            config,
429            backend,
430            lineage_data,
431            non_empty_histories,
432        })
433    }
434}

436/// These methods provide the ability to perform basic operations on the forest without the need to
437/// query the backend.
438///
439/// # Performance
440///
441/// All of these methods can be performed fully in-memory, and hence their performance is
442/// predictable on a given machine regardless of the choice of [`Backend`] instance being used by
443/// the forest.
444impl<B: Backend> LargeSmtForest<B> {
445    /// Returns an iterator that yields all the (uniquely identified) roots that the forest knows
446    /// about, including those from historical versions.
447    ///
448    /// The iteration order of these roots is unspecified.
449    pub fn roots(&self) -> impl Iterator<Item = UniqueRoot> {
450        // As the history container does not deal in roots with domains, we have to attach the
451        // corresponding domain to each root, and do this as lazily as possible to avoid
452        // materializing more things than we need to.
453        self.lineage_data
454            .iter()
455            .flat_map(|(l, d)| d.roots().map(|r| UniqueRoot::new(*l, r)))
456    }
457
458    /// Gets the latest version of the tree for the provided `lineage`, if that lineage is in the
459    /// forest, or returns [`None`] otherwise.
460    pub fn latest_version(&self, lineage: LineageId) -> Option<VersionId> {
461        self.lineage_data.get(&lineage).map(|d| d.latest_version)
462    }
463
464    /// Returns an iterator that yields the root values for trees within the specified `lineage`, or
465    /// [`None`] if the lineage is not known.
466    ///
467    /// The iteration order of the roots is guaranteed to move backward in time as the iterator
468    /// advances, with earlier items being roots from versions closer to the present. The current
469    /// root of the lineage will thus always be the first item yielded by the iterator.
470    pub fn lineage_roots(&self, lineage: LineageId) -> Option<impl Iterator<Item = RootValue>> {
471        self.lineage_data.get(&lineage).map(|d| d.roots())
472    }
473
474    /// Gets the value root of the newest tree in the provided `lineage`, if that lineage is in the
475    /// forest, or returns [`None`] otherwise.
476    pub fn latest_root(&self, lineage: LineageId) -> Option<RootValue> {
477        self.lineage_data.get(&lineage).map(|d| d.latest_root)
478    }
479
480    /// Returns the number of trees in the forest that have unique identity.
481    ///
482    /// This is **not** the number of unique tree lineages in the forest, as it includes all
483    /// historical trees as well. For that, see [`Self::lineage_count`].
484    pub fn tree_count(&self) -> usize {
485        self.roots().count()
486    }
487
488    /// Returns the number of unique tree lineages in the forest.
489    ///
490    /// This is **not** the number of unique trees in the forest, as it does not include all
491    /// versions in each lineage. For that, see [`Self::tree_count`].
492    pub fn lineage_count(&self) -> usize {
493        self.lineage_data.len()
494    }
495
496    /// Returns data describing what information the forest knows about the provided `root`.
497    pub fn root_info(&self, root: TreeId) -> RootInfo {
498        let Some(d) = self.lineage_data.get(&root.lineage()) else {
499            return RootInfo::Missing;
500        };
501
502        if d.latest_version == root.version() {
503            return RootInfo::LatestVersion(d.latest_root);
504        }
505
506        if root.version() > d.latest_version {
507            return RootInfo::Missing;
508        }
509
510        match d.history.root_for_version(root.version()) {
511            Ok(r) => RootInfo::HistoricalVersion(r),
512            Err(_) => RootInfo::Missing,
513        }
514    }
515
516    /// Removes all tree versions in the forest that are older than the provided `version`, but
517    /// always retains the latest tree in each lineage.
518    pub fn truncate(&mut self, version: VersionId) {
519        let mut newly_empty = Set::default();
520
521        self.non_empty_histories.iter().for_each(|l| {
522            if let Some(d) = self.lineage_data.get_mut(l)
523                && d.truncate(version)
524            {
525                newly_empty.insert(*l);
526            }
527        });
528
529        self.non_empty_histories.extend(newly_empty);
530    }
531}

536/// These methods pertain to non-mutating queries about the data stored in the forest. They differ
537/// from the simple queries in the previous block by requiring access to the backend to function.
538///
539/// # Performance
540///
541/// All the methods in this impl block require access to the underlying [`Backend`] instance to
542/// return results. This means that their performance will depend heavily on the specific instance
543/// with which the forest was constructed.
544///
545/// Where anything more specific can be said about performance, the method documentation will
546/// contain more detail.
547impl<B: Backend> LargeSmtForest<B> {
548    /// Returns an opening for the specified `key` in the specified `tree`, regardless of whether
549    /// the `tree` has a value associated with `key` or not.
550    ///
551    /// # Errors
552    ///
553    /// - [`LargeSmtForestError::Fatal`] if the backend fails to operate properly during the query.
554    /// - [`LargeSmtForestError::UnknownLineage`] If the provided `tree` specifies a lineage that is
555    ///   not one known by the forest.
556    /// - [`LargeSmtForestError::UnknownTree`] if the provided `tree` refers to a tree that is not a
557    ///   member of the forest.
558    /// - [`LargeSmtForestError::Merkle`] if there is insufficient data in the specified `tree` to
559    ///   provide an opening for `key`.
560    pub fn open(&self, tree: TreeId, key: Word) -> Result<SmtProof> {
561        // We want to return an error if the lineage is unknown to comply with the stated contract
562        // for the function.
563        let lineage_data = self
564            .lineage_data
565            .get(&tree.lineage())
566            .ok_or(LargeSmtForestError::UnknownLineage(tree.lineage()))?;
567
568        // We then check if the version exists in the forest. We do this before fetching the full
569        // tree as to do so otherwise would represent a possible denial-of-service vector.
570        if tree.version() > lineage_data.latest_version {
571            // Here the tree is newer than we know about, and so we should error.
572            return Err(LargeSmtForestError::UnknownTree(tree));
573        }
574
575        if tree.version() == lineage_data.latest_version {
576            // In this case we can service the opening directly from the backend as the query is for
577            // the latest version of the tree.
578            return self.backend.open(tree.lineage(), key).map_err(Into::into);
579        }
580
581        let Ok(view) = lineage_data.history.get_view_at(tree.version()) else {
582            // In this case, either the version in `tree` is newer than the latest we know about, so
583            // we can't provide an opening, or it is not serviceable by the history. In either case,
584            // the specified tree is unknown to the forest.
585            return Err(LargeSmtForestError::UnknownTree(tree));
586        };
587
588        // We start by computing the relevant leaf index and getting the opening from the full
589        // tree to do our (potentially) most-expensive work up front.
590        let leaf_index = LeafIndex::from(key);
591        let opening = self
592            .backend
593            .open(tree.lineage(), key)
594            .map_err(Into::<LargeSmtForestError>::into)?;
595
596        // We compute the new leaf and new path by applying any reversions from the history on
597        // top of the current state.
598        let new_leaf = self.merge_leaves(opening.leaf(), &view.leaf_delta(&leaf_index))?;
599        let new_path = self.merge_paths(leaf_index, opening.path(), view)?;
600
601        // Finally we can compose our combined opening.
602        Ok(SmtProof::new(new_path, new_leaf)?)
603    }
604
605    /// Returns the value associated with the provided `key` in the specified `tree`, or [`None`] if
606    /// there is no non-default value corresponding to the provided `key` in that tree.
607    ///
608    /// # Errors
609    ///
610    /// - [`LargeSmtForestError::Fatal`] if the backend fails to operate properly during the query.
611    /// - [`LargeSmtForestError::UnknownLineage`] if the provided `tree` specifies a lineage that is
612    ///   not one known by the forest.
613    /// - [`LargeSmtForestError::UnknownTree`] if the provided `tree` refers to a tree that is not a
614    ///   member of the forest.
615    pub fn get(&self, tree: TreeId, key: Word) -> Result<Option<Word>> {
616        // We want to return an error if the lineage is unknown to comply with the stated contract
617        // for the function.
618        let lineage_data = self
619            .lineage_data
620            .get(&tree.lineage())
621            .ok_or(LargeSmtForestError::UnknownLineage(tree.lineage()))?;
622
623        if tree.version() > lineage_data.latest_version {
624            // Here the tree is newer than we know about, and so we should error.
625            return Err(LargeSmtForestError::UnknownTree(tree));
626        }
627
628        if tree.version() == lineage_data.latest_version {
629            // In this case we can service the opening directly from the backend as the query is for
630            // the latest version of the tree.
631            return self.backend.get(tree.lineage(), key).map_err(Into::into);
632        }
633
634        let Ok(view) = lineage_data.history.get_view_at(tree.version()) else {
635            // In this case, either the version in `tree` is newer than the latest we know about, so
636            // we can't provide an opening, or it is not serviceable by the history. In either case,
637            // the specified tree is unknown to the forest.
638            return Err(LargeSmtForestError::UnknownTree(tree));
639        };
640
641        // We prioritize the value in the history if one exists, falling back to the full tree
642        // if none does. We don't use `or` here because we don't want to query the backend
643        // unless we have to, and we can't use `or_else` due to lack of support for `Result`.
644        let result = if let Some(value) = view.value(&key) {
645            // If the history value is an empty word, the value was unset in the historical tree
646            // version, so we have to conform to our interface by returning `None` here.
647            if value == EMPTY_WORD { None } else { Some(value) }
648        } else {
649            self.backend.get(tree.lineage(), key)?
650        };
651
652        // We can just return that directly.
653        Ok(result)
654    }
655
656    /// Returns the number of populated entries in the specified `tree`.
657    ///
658    /// # Performance
659    ///
660    /// Due to the way that tree data is stored, this method exhibits a split performance profile.
661    ///
662    /// - If querying for a `tree` that is the latest in its lineage, the time to return a result
663    ///   should be constant.
664    /// - If querying for a `tree` that is a historical version, the time to return a result will be
665    ///   linear in the number of entries in the tree. This is because an overlaid iterator has to
666    ///   be created to yield the correct entries for the historical version, and then queried for
667    ///   its length.
668    ///
669    /// # Errors
670    ///
671    /// - [`LargeSmtForestError::Fatal`] if the backend fails to operate properly during the query.
672    /// - [`LargeSmtForestError::UnknownLineage`] if the provided `tree` specifies a lineage that is
673    ///   not one known by the forest.
674    /// - [`LargeSmtForestError::UnknownTree`] if the provided `tree` refers to a tree that is not a
675    ///   member of the forest.
676    pub fn entry_count(&self, tree: TreeId) -> Result<usize> {
677        // We start by yielding an error if we cannot get the lineage data for the specified tree.
678        let Some(lineage_data) = self.lineage_data.get(&tree.lineage()) else {
679            return Err(LargeSmtForestError::UnknownLineage(tree.lineage()));
680        };
681
682        if tree.version() > lineage_data.latest_version {
683            // Here the tree is newer than we know about, and so we should error.
684            return Err(LargeSmtForestError::UnknownTree(tree));
685        }
686
687        if tree.version() == lineage_data.latest_version {
688            // We can fast-path the current tree using the backend.
689            return Ok(self.backend.entry_count(tree.lineage())?);
690        }
691
692        let Ok(view) = lineage_data.history.get_view_at(tree.version()) else {
693            // If neither of these are the case, we do not know the version and so fail out.
694            return Err(LargeSmtForestError::UnknownTree(tree));
695        };
696
697        // In the general case there is no faster path than doing the iteration to merge the
698        // history with the full tree, so we just count the iterator.
699        Ok(
700            EntriesIterator::new_with_history(
701                self.backend.entries(tree.lineage())?,
702                view.entries(),
703            )
704            .count(),
705        )
706    }
707
708    /// Returns an iterator that yields the entries in the specified `tree`.
709    ///
710    /// # Performance
711    ///
712    /// The performance of the iterator depends both on the choice of backend _and_ the type of tree
713    /// that is queried for. We cannot give exact performance figures, but in general querying over
714    /// **the current tree** in a lineage will be faster than querying over **a historical tree** in
715    /// a lineage.
716    ///
717    /// # Errors
718    ///
719    /// - [`LargeSmtForestError::Fatal`] if the backend fails to operate properly during the query.
720    /// - [`LargeSmtForestError::UnknownLineage`] if the provided `tree` specifies a lineage that is
721    ///   not one known by the forest.
722    /// - [`LargeSmtForestError::UnknownTree`] if the provided `tree` refers to a tree that is not a
723    ///   member of the forest.
724    pub fn entries(&self, tree: TreeId) -> Result<impl Iterator<Item = TreeEntry>> {
725        // We start by yielding an error if we cannot get the lineage data for the specified tree.
726        let Some(lineage_data) = self.lineage_data.get(&tree.lineage()) else {
727            return Err(LargeSmtForestError::UnknownLineage(tree.lineage()));
728        };
729
730        if tree.version() > lineage_data.latest_version {
731            // Here the tree is newer than we know about, and so we should error.
732            return Err(LargeSmtForestError::UnknownTree(tree));
733        }
734
735        if tree.version() == lineage_data.latest_version {
736            // If we match the current version, we can construct the simple iterator variant.
737            return Ok(EntriesIterator::new_without_history(self.backend.entries(tree.lineage())?));
738        }
739
740        let Ok(view) = lineage_data.history.get_view_at(tree.version()) else {
741            // If neither of these are the case, we do not know the version and so fail out.
742            return Err(LargeSmtForestError::UnknownTree(tree));
743        };
744
745        // If we can serve it from the history we need to instead construct the complex version.
746        Ok(EntriesIterator::new_with_history(
747            self.backend.entries(tree.lineage())?,
748            view.entries(),
749        ))
750    }
751}

753// SINGLE-TREE MODIFIERS
754// ================================================================================================
755
756/// These methods pertain to modifications that can be made to a single tree in the forest. They
757/// exploit parallelism within the single target tree wherever possible.
758///
759/// # Performance
760///
761/// All the methods in this impl block require access to the underlying [`Backend`] instance to
762/// return results. This means that their performance will depend heavily on the specific instance
763/// with which the forest was constructed.
764///
765/// Where anything more specific can be said about performance, the method documentation will
766/// contain more detail.
767impl<B: Backend> LargeSmtForest<B> {
768    /// Adds a new `lineage` to the tree, creating an empty tree and modifying it as specified by
769    /// `updates`, with the result taking the provided `new_version`.
770    ///
771    /// If the provided `updates` batch is empty, then the **empty tree will be added** as the first
772    /// version in the lineage.
773    ///
774    /// # Errors
775    ///
776    /// - [`LargeSmtForestError::DuplicateLineage`] if the provided `lineage` is the same as an
777    ///   already-known lineage.
778    /// - [`LargeSmtForestError::Fatal`] if the backend fails while being accessed.
779    /// - [`BackendError::Merkle`] if the provided `updates` cannot be applied to the empty tree.
780    pub fn add_lineage(
781        &mut self,
782        lineage: LineageId,
783        new_version: VersionId,
784        updates: SmtUpdateBatch,
785    ) -> Result<TreeWithRoot> {
786        // We can immediately add lineage in the backend, as by its contract it should return
787        // `DuplicateLineage` if the new lineage is a duplicate. We forward that, and any other
788        // errors, as this is the correct behavior for correctly-implemented backends.
789        let tree_info = self.backend.add_lineage(lineage, new_version, updates)?;
790
791        // We then construct the lineage tracking data and shove it into the corresponding map. The
792        // history is guaranteed to be empty here, so we do not need to put an entry in the
793        // non-empty histories set.
794        let lineage_data = LineageData {
795            history: History::empty(self.config.max_history_versions()),
796            latest_version: tree_info.version(),
797            latest_root: tree_info.root(),
798        };
799        self.lineage_data.insert(lineage, lineage_data);
800
801        Ok(tree_info)
802    }
803
804    /// Performs the provided `updates` on the latest tree in the specified `lineage`, adding a
805    /// single new root to the forest (corresponding to `new_version`) for the entire batch, and
806    /// returning the data for the new root of the tree.
807    ///
808    /// If applying the provided `operations` results in no changes to the tree, then the root data
809    /// will be returned unchanged and no new tree will be allocated. It will retain its original
810    /// version, and not be returned with `new_version`.
811    ///
812    /// # Errors
813    ///
814    /// - [`LargeSmtForestError::BadVersion`] if the `new_version` is older than the latest version
815    ///   for the provided `lineage`.
816    /// - [`LargeSmtForestError::Fatal`] if the backend fails while being accessed.
817    /// - [`LargeSmtForestError::UnknownLineage`] if the provided `tree` specifies a lineage that is
818    ///   not one known by the forest.
819    pub fn update_tree(
820        &mut self,
821        lineage: LineageId,
822        new_version: VersionId,
823        updates: SmtUpdateBatch,
824    ) -> Result<TreeWithRoot> {
825        // We initially check that the lineage is known and that the version is greater than the
826        // last known version for that lineage.
827        let lineage_data = if let Some(lineage_data) = self.lineage_data.get_mut(&lineage) {
828            if lineage_data.latest_version < new_version {
829                lineage_data
830            } else {
831                return Err(LargeSmtForestError::BadVersion {
832                    provided: new_version,
833                    latest: lineage_data.latest_version,
834                });
835            }
836        } else {
837            return Err(LargeSmtForestError::UnknownLineage(lineage));
838        };
839
840        // We now know that we have a valid lineage and a valid version, so we perform the update in
841        // the backend.
842        let reversion_set = self.backend.update_tree(lineage, new_version, updates)?;
843
844        // We do not want to actually change anything if the tree would not change.
845        if reversion_set.is_empty() {
846            return Ok(TreeWithRoot::new(
847                lineage,
848                lineage_data.latest_version,
849                lineage_data.latest_root,
850            ));
851        }
852
853        // The new root of the latest tree is actually given by the **old root** in our reverse
854        // mutation set.
855        let updated_root = reversion_set.old_root;
856
857        // The call to `add_version_from_mutation_set` should only yield an error if the
858        // provided version does not pass the version check. This check has already been
859        // performed as a precondition for reaching this point of the tree update, and
860        // hence should only ever fail due to a programmer bug so we panic if it does fail.
861        lineage_data
862            .history
863            .add_version_from_mutation_set(lineage_data.latest_version, reversion_set)
864            .unwrap_or_else(|_| {
865                panic!("Unable to add valid version {} to history", lineage_data.latest_version)
866            });
867
868        // At this point we now have a historical version added, so we track that the lineage has a
869        // non-empty history.
870        self.non_empty_histories.insert(lineage);
871
872        // Now we just have to update the other portions of the lineage data in place...
873        lineage_data.latest_root = updated_root;
874        lineage_data.latest_version = new_version;
875
876        // ...and return the correct value.
877        Ok(TreeWithRoot::new(lineage, new_version, updated_root))
878    }
879}

881// MULTI-TREE MODIFIERS
882// ================================================================================================
883
884/// These methods pertain to modifications that can be made to multiple trees in the forest at once.
885/// They exploit parallelism both between trees and within trees wherever possible.
886///
887/// # Performance
888///
889/// All the methods in this impl block require access to the underlying [`Backend`] instance to
890/// return results. This means that their performance will depend heavily on the specific instance
891/// with which the forest was constructed.
892///
893/// Where anything more specific can be said about performance, the method documentation will
894/// contain more detail.
895impl<B: Backend> LargeSmtForest<B> {
896    /// Performs the provided `updates` on the forest, adding at most one new root with version
897    /// `new_version` to the forest for each target root in `updates` and returning a mapping
898    /// from old root to the new root data.
899    ///
900    /// If applying the associated batch to any given lineage in the forest results in no changes to
901    /// that tree, the initial root for that lineage will be returned and no new tree will be
902    /// allocated.
903    ///
904    /// # Errors
905    ///
906    /// - [`LargeSmtForestError::UnknownLineage`] If any lineage in the batch of modifications is
907    ///   one that is not known by the forest.
908    /// - [`LargeSmtForestError::Fatal`] if any error occurs to leave the forest in an inconsistent
909    ///   state.
910    /// - [`LargeSmtForestError::BadVersion`] if the `new_version` is older than the latest version
911    ///   for the provided `lineage`.
912    pub fn update_forest(
913        &mut self,
914        new_version: VersionId,
915        updates: SmtForestUpdateBatch,
916    ) -> Result<Vec<TreeWithRoot>> {
917        // We start by performing our precondition checks on the lineages and versions. We have to
918        // ensure both that all the lineages exist, and that the specified version transition is
919        // valid for all of those lineages.
920        updates
921            .lineages()
922            .map(|lineage| {
923                let Some(lineage_data) = self.lineage_data.get(lineage) else {
924                    return Err(LargeSmtForestError::UnknownLineage(*lineage));
925                };
926
927                if lineage_data.latest_version < new_version {
928                    Ok(())
929                } else {
930                    Err(LargeSmtForestError::BadVersion {
931                        provided: new_version,
932                        latest: lineage_data.latest_version,
933                    })
934                }
935            })
936            .collect::<Result<Vec<_>>>()?;
937
938        // With the preconditions checked we can call into the backend to perform the updates, and
939        // we forward all errors as this will be correct for conformant backend implementations.
940        let reversion_sets = self.backend.update_forest(new_version, updates)?;
941
942        // Now we have to update the lineage data (including the history) to ensure that the state
943        // remains consistent, and we build our return values while doing so.
944        reversion_sets
945            .into_iter()
946            .map(|(lineage, reversion)| {
947                let lineage_data = self
948                    .lineage_data
949                    .get_mut(&lineage)
950                    .expect("Lineage has been checked to be present");
951
952                // If the operations change nothing we want to short-circuit for that tree.
953                if reversion.is_empty() {
954                    return Ok(TreeWithRoot::new(
955                        lineage,
956                        lineage_data.latest_version,
957                        lineage_data.latest_root,
958                    ));
959                }
960
961                let updated_root = reversion.old_root;
962
963                // The call to `add_version_from_mutation_set` should only yield an error if the
964                // provided version does not pass the version check. This check has already been
965                // performed as a precondition for reaching this point of the forest update, and
966                // hence should only ever fail due to a programmer bug so we panic if it does fail.
967                lineage_data
968                    .history
969                    .add_version_from_mutation_set(lineage_data.latest_version, reversion)
970                    .unwrap_or_else(|_| {
971                        panic!(
972                            "Unable to add valid version {} to history",
973                            lineage_data.latest_version
974                        )
975                    });
976
977                // At this point we know that we have a historical version for that tree, so we
978                // should track it as having a non-empty history.
979                self.non_empty_histories.insert(lineage);
980
981                lineage_data.latest_root = updated_root;
982                lineage_data.latest_version = new_version;
983
984                Ok(TreeWithRoot::new(lineage, new_version, updated_root))
985            })
986            .collect::<Result<Vec<_>>>()
987    }
988}

990// INTERNAL UTILITY FUNCTIONS
991// ================================================================================================
992
993/// This block contains internal functions that exist to de-duplicate or modularize functionality
994/// within the forest. These should not be exposed.
995impl<B: Backend> LargeSmtForest<B> {
996    /// Applies the provided `historical_delta` on top of the provided `full_tree_leaf` to produce
997    /// the correct leaf for a historical opening.
998    fn merge_leaves(
999        &self,
1000        full_tree_leaf: &SmtLeaf,
1001        historical_delta: &CompactLeaf,
1002    ) -> Result<SmtLeaf> {
1003        // We apply the historical delta on top of the existing entries to perform the reversion
1004        // back to the previous state.
1005        let mut leaf_entries = Map::new();
1006        leaf_entries.extend(full_tree_leaf.to_entries().map(|(k, v)| (*k, *v)));
1007        leaf_entries.extend(historical_delta);
1008
1009        // Any entries that are still empty at this point should be removed.
1010        let non_empties_only = leaf_entries.into_iter().filter(|(_, v)| *v != EMPTY_WORD).collect();
1011        Ok(SmtLeaf::new(non_empties_only, full_tree_leaf.index())?)
1012    }
1013
1014    /// Applies any historical changes contained in `history_view` on top of the merkle path
1015    /// obtained from the full tree to produce the correct path for a historical opening.
1016    fn merge_paths(
1017        &self,
1018        leaf_index: LeafIndex<SMT_DEPTH>,
1019        full_tree_path: &SparseMerklePath,
1020        history_view: HistoryView,
1021    ) -> Result<SparseMerklePath> {
1022        let mut path_elems = [EMPTY_WORD; SMT_DEPTH as usize];
1023        let mut current_node_ix = NodeIndex::from(leaf_index);
1024        for depth in (1..=SMT_DEPTH).rev() {
1025            // This is the sibling node of the currently-tracked node. In other words, it is the
1026            // node that needs to become part of the path.
1027            let path_node_ix = current_node_ix.sibling();
1028
1029            if let Some(historical_value) = history_view.node_value(&path_node_ix) {
1030                // If there is a historical value we need to use it, and so we write it to the
1031                // correct slot in the path elements array.
1032                path_elems[depth as usize - 1] = *historical_value;
1033            } else {
1034                // If there isn't a historical value, we should delegate to the corresponding
1035                // element in the path from the full-tree opening.
1036                let bounded_depth = NonZeroU8::new(depth).expect("depth  1 ..= SMT_DEPTH]");
1037                path_elems[depth as usize - 1] = full_tree_path.at_depth(bounded_depth)?
1038            }
1039
1040            // We then need to move upward in the tree of the nodes we know.
1041            current_node_ix = current_node_ix.parent();
1042        }
1043
1044        // Now that we have filled in our `path_elems` we can use the construction of a sparse
1045        // merkle path from a sized iterator, and thus not compute the mask ourselves. We
1046        // reverse the iterator to make it go from deepest to shallowest as required.
1047        Ok(SparseMerklePath::from_sized_iter(path_elems.into_iter().rev())?)
1048    }
1049}
